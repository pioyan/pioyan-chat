# TDD セクションテンプレート

コード実装を行うエージェントに組み込む TDD（テスト駆動開発）セクションの独立リファレンスです。
`@agent-builder` が新規エージェント生成時に参照し、コード実装エージェントに TDD セクションを組み込みます。

## TDD 適用判断フロー

```text
エージェントはコードを実装するか？
│
├─ はい → TDD セクションを組み込む
│   │
│   ├─ テストフレームワークが指定されているか？
│   │   ├─ はい → 指定のフレームワークを使用
│   │   └─ いいえ → 言語別デフォルト（下記テーブル）を採用
│   │
│   └─ ツールに runInTerminal を含める（テスト実行用）
│
└─ いいえ → TDD セクションは省略
    └─ 監査・レビュー・ドキュメント系エージェント
```

### 「コード実装を行う」の判断基準

以下のいずれかに該当する場合、コード実装エージェントとみなす:

- アプリケーションコード（関数・クラス・モジュール）を新規作成する
- 既存のアプリケーションコードを修正・リファクタリングする
- テストコードを新規作成する

以下は該当しない:

- 設定ファイル（YAML・JSON・TOML）の生成・編集のみ
- ドキュメント（Markdown）の作成・更新のみ
- CI ワークフローの構成のみ

## TDD セクション本文

以下のセクションを、エージェント定義の `## 禁止事項` の直前に挿入してください。

````markdown
## TDD 実装サイクル（t-wada 式）

コードの実装は必ず以下の TDD サイクルに従って進める。

### Red — 失敗するテストを書く

1. 実装したい振る舞いを **1 つだけ** 選ぶ
2. その振る舞いを検証するテストを書く
3. テストを実行し、**失敗することを確認** する（テスト自体の正しさの検証）
4. テストが正しく要件を表現していることを確認する

### Green — テストを通す最小限のコードを書く

1. テストを通すための **最小限のコード** だけを書く
2. ハードコードや仮実装（Fake it）でもよい
3. テストを実行し、**成功することを確認** する
4. 過剰な実装・先回りした設計をしない

### Refactor — コードを改善する

1. テストが通っている状態を維持しつつリファクタリングする
2. 重複の除去、命名の改善、責務の分離を行う
3. リファクタリングのたびにテストを実行し、すべて通ることを確認する
4. テストコード自体のリファクタリングも行う

### サイクルの原則

- **1 サイクルの粒度**: 数分〜十数分。小さいステップを積み重ねる
- **動作するコードを常に保つ**: リファクタリング中にテストが壊れたら即座に戻す
- **テストなしのコード変更は行わない**: 新機能・バグ修正・リファクタリングすべてテストが先
- **三角測量**: 1 つのテストでは仮実装で通るなら、2 つ目のテストを追加して実装を駆動する
````

## 言語別テスト実行コマンド

エージェントに TDD セクションを組み込む際、技術スタックに応じたテストコマンドを
作業手順のテスト実行ステップに記載してください。

| 言語 | テストフレームワーク | テスト実行コマンド | ウォッチモード |
|------|-------------------|------------------|--------------|
| Node.js | Vitest | `npx vitest run` | `npx vitest` |
| Node.js | Jest | `npx jest` | `npx jest --watch` |
| Python | pytest | `pytest` | `pytest-watch` (`ptw`) |
| Go | go test | `go test ./...` | — |
| Rust | cargo test | `cargo test` | `cargo watch -x test` |
| Java | JUnit (Maven) | `mvn test` | — |
| Java | JUnit (Gradle) | `gradle test` | `gradle test --continuous` |

## TDD セクション挿入位置

エージェント定義内での挿入位置:

```text
# エージェントタイトル

## 行動原則
## ヒアリング項目（対話生成型のみ）
## チェックリスト（任意）
## 作業手順
## TDD 実装サイクル（t-wada 式）  ← ここに挿入
## 禁止事項
```

## TDD とエージェント作業手順の統合例

コード実装エージェントの `## 作業手順` には、TDD サイクルを組み込んだステップを記載してください:

```markdown
## 作業手順

### Step 1: 要件確認
ユーザーから実装する機能の要件を確認する。

### Step 2: テスト環境の準備
テストフレームワークの設定を確認し、テストが実行できる状態にする。

### Step 3: TDD サイクルで実装
以下のサイクルを繰り返して実装を進める:

1. **Red**: 次に実装する振る舞いのテストを 1 つ書き、失敗を確認
2. **Green**: テストを通す最小限のコードを書き、成功を確認
3. **Refactor**: コードを改善し、テストがすべて通ることを確認

### Step 4: 最終検証
すべてのテストが通ることを確認し、カバレッジを確認する。

### Step 5: 完了報告
実装したファイルの一覧とテスト結果を報告する。
```
