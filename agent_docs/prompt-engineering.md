# プロンプトエンジニアリングガイド

エージェント定義（`.github/agents/*.md`）やスキル定義（`.github/skills/*/SKILL.md`）を
記述する際に適用すべきベストプラクティス集です。

---

## 1. ロール定義パターン

### 原則

エージェントの冒頭で **誰であるか** と **何を専門とするか** を明示的に宣言する。
LLM はロール設定に基づいてトーンと専門性を調整するため、具体的であるほど出力品質が向上する。

### テンプレート

```markdown
あなたは **<Agent Name>** です。<対象ドメイン>の<具体的なタスク>を行う専門エージェントです。
```

### 正例

```markdown
あなたは **Code Reviewer** です。Pull Request の差分を分析し、
品質・セキュリティ・保守性の観点からレビューコメントを提案する専門エージェントです。
```

### 反例

```markdown
あなたはコードレビューを手伝うアシスタントです。
```

> **なぜ反例が悪いか**: 「手伝う」「アシスタント」は役割が曖昧。
> 「差分を分析し」「レビューコメントを提案する」のように具体的な行動を含めること。

## 2. 構造化指示

### 原則

Markdown の見出し階層を活用して、指示を論理的なセクションに分割する。
LLM は構造化された入力を正確に処理できるため、フラットな箇条書きより見出し付きの構造が有効。

### 推奨構造

```text
# タイトル — サブタイトル        ← H1: エージェント名と役割（1 つのみ）

ロール定義文

## 行動原則                      ← H2: 行動の基本方針
## チェックリスト                ← H2: タスクリスト形式の確認項目
### カテゴリ名                   ← H3: チェックリストの分類
## 作業手順                      ← H2: 実行ステップ
### Step 1: タスク名             ← H3: 各ステップの詳細
## 禁止事項                      ← H2: やってはいけないこと
```

### 正例

```markdown
## レビュー観点チェックリスト

### セキュリティ
- [ ] ハードコードされたシークレットがないか
- [ ] ユーザー入力のバリデーションが適切か

### コード品質
- [ ] 命名が意図を反映しているか
- [ ] 重複コードがないか
```

### 反例

```markdown
## チェックリスト
- セキュリティチェック
- コード品質チェック
- テストチェック
- ドキュメントチェック
```

> **なぜ反例が悪いか**: カテゴリが平坦で具体的な確認項目がない。
> チェックリスト形式（`- [ ]`）で個別項目に分解し、H3 でカテゴリを分けること。

## 3. 制約の明示化

### 原則

エージェントが**やるべきこと**と**やってはいけないこと**を明確に分離する。
肯定形の指示（行動原則）を先に、否定形の指示（禁止事項）を後に配置する。

### 行動原則の書き方

番号付きリスト + 太字キーワード + コロン + 説明文:

```markdown
## 行動原則

1. **最小差分**: 既存のファイルやスタイルを尊重し、不足分だけを追加する
2. **安全第一**: シークレットや認証情報を出力しない
3. **透明性**: 変更理由を必ず記載する
```

### 禁止事項の書き方

箇条書き + 具体的な禁止行為:

```markdown
## 禁止事項

- アプリケーションコードの直接変更（設定・ドキュメントのみ対象）
- 既存ファイルの無断上書き（必ず差分を提示し確認を取ること）
- シークレット・トークン・パスワードの出力
```

### 括弧内の補足で理由や代替を示す

```markdown
- PR の自動マージや Approve の実行（提案のみ）
```

> 「提案のみ」が代替行動を示しており、エージェントは何をすべきかが明確になる。

## 4. チェックリスト活用

### 原則

進捗追跡が可能なタスクリスト形式（`- [ ]`）を使い、エージェントが各項目を
順に確認・完了できるようにする。

### 使い分け

| 形式 | 用途 | 例 |
|------|------|-----|
| `- [ ]` タスクリスト | 実行時に順次チェックする項目 | 監査チェックリスト、生成対象リスト |
| `1. 2. 3.` 番号リスト | 順序が重要な手順 | 行動原則、作業ステップ |
| `-` 箇条書き | 順序不問の列挙 | 禁止事項、参考情報 |

### 正例

```markdown
## 監査対象チェックリスト

### コミュニティヘルスファイル
- [ ] `README.md` — プロジェクト概要・セットアップ手順・ライセンス表記
- [ ] `CONTRIBUTING.md` — 開発参加手順・コーディング規約
- [ ] `SECURITY.md` — 脆弱性報告の導線
```

> ファイル名 + ダッシュ + 確認内容の形式で、何を確認すべきか具体的に記述する。

## 5. Few-shot テンプレート

### 原則

期待する入出力の具体例を含めることで、エージェントの出力品質と一貫性を向上させる。
特に出力フォーマットが定まっている場合に有効。

### 正例（出力フォーマットの定義）

````markdown
## 出力フォーマット

```markdown
## コードレビューサマリー

**PR**: #{PR番号} {タイトル}
**評価**: {Approve / Request Changes / Comment}

### 要対応（セキュリティ・バグ）
- `{ファイル}:{行}` — {指摘内容}

### 改善推奨
- `{ファイル}:{行}` — {指摘内容}

### 良い点
- {特筆すべき改善点}
```
````

### テンプレート変数の記法

| 記法 | 用途 | 例 |
|------|------|-----|
| `{変数名}` | 単一値の挿入 | `{PR番号}`, `{ファイル}` |
| `{値1 / 値2 / 値3}` | 選択肢の列挙 | `{Approve / Request Changes / Comment}` |
| `<kebab-case-name>` | 命名規則に従う値 | `<name>`, `<short-description>` |

## 6. 段階的指示

### 原則

作業手順は `### Step N: タスク名` 形式で段階的に記述する。
各ステップは独立して理解・実行できるようにし、前のステップの完了を前提とする。

### 正例

```markdown
## 作業手順

### Step 1: ヒアリング
ユーザーから要件を収集する。必須項目が揃うまで次のステップに進まない。

### Step 2: 現状把握
リポジトリの既存ファイルを確認する。

### Step 3: 設計提案
収集した情報から設計を作成し、ユーザーに提示する。

### Step 4: ファイル生成
承認後、ファイルを生成する。

### Step 5: 検証
生成したファイルの構文チェックを実行する。
```

### 反例

```markdown
## 作業手順

ユーザーの要件を聞いて、既存ファイルを確認してから、設計して承認をもらってファイルを作ります。
```

> **なぜ反例が悪いか**: すべてが一文に詰め込まれており、各ステップの境界が不明確。
> エージェントがどこまで完了したかの追跡ができない。

## 7. スコープ制限

### 原則

エージェントの責務範囲を明確に限定する。「何をするか」だけでなく「何をしないか」を明示する。

### パターン

```markdown
## 禁止事項

- アプリケーションコードの修正（ドキュメント・設定ファイルのみ変更すること）
```

括弧内で「代わりに何をするか」を示すことで、エージェントが正しい行動にフォールバックできる。

### エージェント間の分業

```text
@repo-guardian  → 設定・テンプレートの自動追加（コードは変更しない）
@code-reviewer  → コードの分析・提案（変更は実行しない）
@docs-writer    → ドキュメントの更新（コードは変更しない）
```

---

## 8. 検証ステップの組み込み

### 原則

ファイル生成後に構文チェックや lint を実行するステップを作業手順に明示的に含める。
「検証する」ではなく具体的なコマンドを記載する。

### 正例

```markdown
### Step 5: 検証

生成したファイルの構文チェックを実行する:

- YAML: `yamllint <file>`
- Markdown: `markdownlint-cli2 <file>`
- GitHub Actions: `actionlint <file>`
```

### 反例

```markdown
### Step 5: 検証

ファイルが正しいか確認する。
```

## 9. 参照の明示化

### 原則

他ファイルを参照する場合は、相対パスではなくリポジトリルートからのパスで明示する。
参照の目的（何のために参照するか）も記載する。

### 正例

```markdown
`/dev-env-setup` スキルのヒアリングテンプレートに従い、ユーザーに要件を質問する。
```

```markdown
## 参照ドキュメント

詳細な設計指針が必要な場合は以下を参照してください:

- `agent_docs/design-patterns.md` — エージェント設計の 3 パターン
- `agent_docs/naming-and-format.md` — 命名規則とフォーマット仕様
```

### 反例

```markdown
他のドキュメントも参考にしてください。
```

> **なぜ反例が悪いか**: どのドキュメントを参照すべきか不明。
> エージェントはファイルパスがなければ参照できない。

## 10. 情報密度の最適化

### 原則

エージェント定義のコンテキストウィンドウは有限であるため、情報密度を高く保つ。
冗長な説明は避け、テーブル・リスト・コードブロックを活用する。

### テーブルの活用

ヒアリング項目や選択肢をテーブルで簡潔にまとめる:

```markdown
| # | 項目 | 例 |
|---|------|-----|
| 1 | プログラミング言語 | Node.js / Python / Go |
| 2 | バージョン | Node.js 22 / Python 3.13 |
| 3 | パッケージマネージャ | npm / pip / cargo |
```

### 条件付きセクションの扱い

任意のセクションは見出しに注釈を添える:

```text
## ヒアリング項目        ← 対話生成型のみ（任意）
## TDD 実装サイクル      ← コード実装エージェントのみ（任意）
```

## チェックリスト: プロンプト品質の自己評価

エージェント定義やスキル定義を作成・レビューする際に確認してください:

- [ ] ロール定義が具体的で、対象ドメインと行動が明記されている
- [ ] 見出し階層が論理的に構造化されている
- [ ] 行動原則（肯定形）と禁止事項（否定形）が分離されている
- [ ] チェックリスト項目にファイル名や確認内容が具体的に記載されている
- [ ] 作業手順が Step 形式で段階的に記述されている
- [ ] 出力フォーマットがテンプレートとして定義されている（読取専用型の場合）
- [ ] 他ファイルへの参照がリポジトリルートからのパスで記載されている
- [ ] 禁止事項に代替行動が括弧内で示されている
- [ ] 情報がテーブル・リスト・コードブロックで効率的に表現されている
- [ ] コード実装エージェントには TDD セクションが組み込まれている
