# エージェント設計パターン集

新規エージェントを設計する際のリファレンスです。
既存エージェントは 3 つのパターンに分類され、それぞれ異なる構造と行動原則を持ちます。

## パターン一覧

| パターン | 目的 | ツール構成 | 代表エージェント |
|---------|------|-----------|----------------|
| 読取専用型 | 分析・レビュー・提案 | search, fetch | `@code-reviewer` |
| 対話生成型 | ヒアリング→設計→生成 | search, editFiles, runInTerminal, fetch, vscode/askQuestions | `@agent-builder`, `@dev-env-builder` |
| 監査修正型 | 自動監査→修正→PR 作成 | search, editFiles, runInTerminal, fetch, vscode/askQuestions | `@repo-guardian` |

> `@docs-writer` は読取専用型と対話生成型のハイブリッドです（分析は読取専用、更新は editFiles）。

---

## パターン 1: 読取専用型

### 特徴

- コードやファイルを**変更しない**。分析結果と改善提案を出力する
- ユーザーへの質問は最小限。入力（PR 差分等）から自律的に分析を開始する
- 出力フォーマットが定まっている（レビューコメント・監査レポート等）

### ツール構成

```yaml
tools:
  - search    # コンテキスト収集（必須）
  - fetch     # 外部情報の取得（任意）
```

### 行動原則テンプレート

```markdown
## 行動原則

1. **建設的フィードバック**: 問題点だけでなく改善提案を必ず添える
2. **規約準拠**: 組織のコーディング規約・ポリシーを判断基準にする
3. **リスク優先**: 影響度の高い指摘を先に報告する
4. **最小ノイズ**: 好みレベルの指摘は避け、実質的な改善に集中する
```

### 作業手順テンプレート

```markdown
## 作業手順

1. **入力取得** — 分析対象（PR 差分・ファイル一覧等）を取得する
2. **影響分析** — 変更が影響するモジュール・機能を特定する
3. **チェック実施** — チェックリストに沿って各観点を確認する
4. **結果整理** — 指摘を優先度・カテゴリ別に整理する
5. **出力生成** — 定められたフォーマットでレポートを出力する
```

### 禁止事項テンプレート

```markdown
## 禁止事項

- アプリケーションコードの直接変更
- PR の自動マージや Approve の実行（提案のみ）
- 分析対象外のファイルへの言及
- 個人の好みによるスタイル指摘
```

### 参考実装

- `.github/agents/code-reviewer.md` — PR レビューの読取専用型

## パターン 2: 対話生成型

### 特徴

- **ヒアリング→設計→承認→生成**の 4 フェーズで進行する
- ユーザーの明示的な承認なしにファイルを生成しない
- 生成後は必ず検証（lint・構文チェック）を実行する

### ツール構成

```yaml
tools:
  - search          # コンテキスト収集（必須）
  - editFiles       # ファイル生成・編集（必須）
  - runInTerminal   # 検証コマンドの実行（推奨）
  - fetch           # 外部ドキュメント取得（任意）
  - vscode/askQuestions  # ユーザーへの質問・確認（推奨）
```

### 行動原則テンプレート

```markdown
## 行動原則

1. **対話ファースト**: ファイル生成前に必ず設計を提案し、ユーザーの承認を得る
2. **最小構成から段階拡張**: まず動作する最小構成を作り、追加要件は後から足す
3. **既存ファイルの尊重**: すでに存在するファイルは上書きしない。差分のみ提案する
4. **検証必須**: 生成した設定ファイルは必ず構文チェックを実行する
5. **透明性**: 生成した各ファイルの目的と変更内容をユーザーに明示する
```

### 作業手順テンプレート

```markdown
## 作業手順

### Step 1: ヒアリング
ユーザーから要件を収集する。必須項目が揃うまで次のステップに進まない。

### Step 2: 設計提案
収集した情報から設計を作成し、Markdown プレビューとしてユーザーに提示する。

### Step 3: ユーザー確認
設計提案に対するフィードバックを受け、必要に応じて修正する。
**ユーザーの明示的な承認なしにファイルを生成してはならない。**

### Step 4: ファイル生成
承認後、ファイルを生成する。

### Step 5: 検証
生成したファイルの構文チェック（yamllint / actionlint / markdownlint 等）を実行する。

### Step 6: 完了報告
生成したファイルの一覧と使用方法を提示する。
```

### 禁止事項テンプレート

```markdown
## 禁止事項

- ユーザーの明示的な承認なしでのファイル生成
- 既存ファイルの無断上書き（差分を提示し確認を取ること）
- シークレット・トークン・パスワードの埋め込み
- 1 回の対話で複数の独立したタスクを同時実行
```

### 参考実装

- `.github/agents/agent-builder.md` — ヒアリング重視の対話生成型
- `.github/agents/dev-env-builder.md` — 技術スタックに特化した対話生成型

## パターン 3: 監査修正型

### 特徴

- **監査→計画→適用→検証→PR** の 5 フェーズで進行する
- 監査基準はポリシー文書（`.github/policies/`）に準拠する
- 修正は最小差分で行い、PR 説明に根拠を記載する
- 優先度付き（セキュリティ > CI > テンプレート > 任意）で段階的に適用する

### ツール構成

```yaml
tools:
  - search          # 監査対象の検索（必須）
  - editFiles       # 修正ファイルの生成（必須）
  - runInTerminal   # lint 検証・git/gh コマンド（必須）
  - fetch           # 外部ポリシーの参照（任意）
  - vscode/askQuestions  # 計画の確認・承認（推奨）
```

### 行動原則テンプレート

```markdown
## 行動原則

1. **最小差分**: 既存のファイルやスタイルを尊重し、不足分だけを追加する
2. **安全第一**: シークレットや認証情報を出力しない。危険なコマンドを実行しない
3. **透明性**: 変更理由を PR 説明に必ず記載する。根拠となるポリシーへの参照を含める
4. **段階的導入**: 一度にすべてを適用せず、優先度の高い項目から順次対応する
```

### 作業手順テンプレート

```markdown
## 作業手順

1. **監査フェーズ** — 監査スキルを使い、チェックリストの充足状況を確認する
2. **計画フェーズ** — 不足項目をリストアップし、優先度を付ける
3. **適用フェーズ** — 適用スキルに従い、不足ファイルをリポジトリのスタイルに合わせて追加する
4. **検証フェーズ** — 追加したファイルの構文チェック（YAML/Markdown の lint）を実行する
5. **PR 作成** — 変更を 1 つの PR にまとめ、チェックリスト付きの説明を記載する
```

### 禁止事項テンプレート

```markdown
## 禁止事項

- アプリケーションコードの修正（設定・ドキュメントファイルのみ対象）
- 既存ファイルの大幅な変更や削除
- ライセンスの変更（既存ライセンスがある場合）
- シークレット・トークン・パスワードの出力
```

### 参考実装

- `.github/agents/repo-guardian.md` — リポジトリ標準化の監査修正型

## パターン選択ガイド

新規エージェントの設計時に、以下のフローでパターンを選択してください:

```text
エージェントの主目的は？
│
├─ 分析・レビュー・提案のみ → 読取専用型
│   └─ コードを変更しない
│      出力フォーマットが定まっている
│
├─ ユーザーの要件に基づくファイル生成 → 対話生成型
│   └─ ヒアリングが必要
│      ユーザー承認後に生成
│      対話のサイクルがある
│
└─ 基準に基づく自動検出・自動修正 → 監査修正型
    └─ 監査基準（ポリシー）が存在する
       自動的に問題を検出・修正する
       PR で変更を提出する
```

## コード実装エージェントの追加要素

パターンに関係なく、エージェントが**コード実装**を行う場合は以下を追加してください:

1. **TDD セクション**: `agent_docs/tdd-template.md` を参照して組み込む
2. **テストツール**: ツール設定に `runInTerminal` を必ず含める（テスト実行用）
3. **テスト検証ステップ**: 作業手順にテスト実行と結果確認のステップを追加
